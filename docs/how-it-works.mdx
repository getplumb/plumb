# How Plumb Works

Plumb is a persistent memory layer for AI agents. It stores high-signal facts across sessions and injects the most relevant ones into every prompt — so your agent remembers context, preferences, and decisions without you having to re-explain them.

## Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│  User ↔ Agent Conversation                                   │
└───────────────────────────┬─────────────────────────────────┘
                            │
                            ▼
              ┌─────────────────────────────┐
              │       Memory Store          │
              │   (SQLite + embeddings)     │
              │                             │
              │  plumb_remember → ingest    │
              │  Session seed at startup    │
              └──────────────┬──────────────┘
                             │
                             ▼
              ┌─────────────────────────────┐
              │      Retrieval Pipeline     │
              ├─────────────────────────────┤
              │  1. BM25 keyword search      │
              │  2. Vector KNN (cosine)      │
              │  3. RRF fusion (k=30)        │
              │  4. Recency decay            │
              │  5. Keyword-overlap rerank   │
              └──────────────┬──────────────┘
                             │
                             ▼
              ┌─────────────────────────────┐
              │   [PLUMB MEMORY] Block      │
              │   Injected before prompt    │
              └─────────────────────────────┘
```

## Memory Storage

Facts are stored as plain text strings with metadata. Each fact contains:

- **Content** — the fact itself, as a plain text string
- **Timestamp** — when it was stored
- **Session ID** — which conversation it came from
- **User ID** — for multi-user setups
- **Vector embedding** — computed locally for semantic search

Facts are written in two ways:

**1. `plumb_remember` tool call**

The agent (or you) explicitly stores a fact. This is the primary write path.

```
plumb_remember("Clay prefers pnpm over npm")
plumb_remember("The deploy target is production-us-east-1")
plumb_remember("Clay's timezone is America/Denver")
```

**2. Session seeding at startup**

On first activation, Plumb seeds facts from local memory files (e.g. `MEMORY.md`, dated daily notes in `memory/`). This bootstraps context from structured notes you already have without re-entering them manually.

## Retrieval Pipeline

When a query arrives (either from prompt injection or an explicit `plumb_search` call), Plumb runs a five-stage hybrid retrieval:

### Stage 1 — BM25 Keyword Search

BM25 (Robertson-Walker variant) with stop word filtering and IDF weighting. Handles exact phrase lookups, names, identifiers, and rare technical terms well.

### Stage 2 — Vector KNN Search

Semantic similarity using local sentence embeddings (Xenova/bge-small-en-v1.5, 384 dimensions). Embeddings are computed in-process via WASM — no external embedding API required. KNN search uses cosine similarity over all stored embeddings.

### Stage 3 — RRF Fusion

Results from BM25 and vector search are merged using Reciprocal Rank Fusion (RRF, k=30). RRF rewards facts that rank well in both keyword and semantic lists, without requiring score normalization between the two.

### Stage 4 — Recency Decay

Each candidate's fused score is multiplied by a decay factor based on age:

```
decay = e^(-λ × age_in_days)
```

Default λ = 0.005, giving a half-life of approximately 140 days. Recent facts rank higher than older ones at equal similarity. This means a closely matching fact from last week naturally scores above the same content from six months ago.

### Stage 5 — Keyword Overlap Reranking

The top candidates receive a final reranking pass that blends the RRF+decay score (70%) with a keyword overlap score between the query and fact content (30%). This improves recall for cases where semantically relevant content uses different phrasing than the query.

### Output

The top-k facts (default: 8) are returned, sorted by final score, and formatted as the `[PLUMB MEMORY]` block.

## Context Injection (OpenClaw Plugin)

In OpenClaw, Plumb runs as a plugin and injects memory automatically before every agent response:

1. User sends a message
2. Plumb intercepts before the model sees it
3. The retrieval pipeline runs with the user message as query
4. Top facts are formatted as a `[PLUMB MEMORY]` block
5. Block is prepended to the system prompt
6. Agent responds with memory context available

**Example injection:**

```
[PLUMB MEMORY]

## Remembered facts
Clay prefers pnpm over npm (3 days ago)
Deploy target is production-us-east-1 (1 week ago)
Clay's timezone is America/Denver (2 weeks ago)

[/PLUMB MEMORY]
```

The injection runs within an 800ms budget. If retrieval takes longer, the prompt proceeds without injection rather than delaying the response.

---

## Explicit Search (plumb_search)

The agent can also run retrieval on demand using `plumb_search`. This uses the same pipeline as auto-injection but lets the agent ask for specific context before acting on something:

```
plumb_search("what package manager does Clay prefer?")
plumb_search("deploy configuration for the API service")
```

In MCP mode (Claude Desktop, Cursor, etc.), the agent decides when to call `plumb_search` explicitly rather than receiving automatic injection.

## Storage

All data is stored in SQLite (WASM build — no native extensions, no install scripts required):

**Database path**: `~/.plumb/memory.db` (default, configurable via `dbPath`)

**Tables**:
- `memory_facts` — stored facts (content, timestamp, session ID, user ID)
- `vec_raw_log` — vector embeddings for facts
- `nudge_log` — internal event log

**Performance at typical scale**:
- Embedding generation: 100–400ms per query (WASM, cold on first query)
- KNN cosine search: ~10ms over thousands of facts (pure JS in-memory loop)
- End-to-end retrieval: 150–500ms at normal DB sizes

---

## Privacy

All memory and embeddings are stored locally on your machine. No cloud access required for normal use:

- **Search** — fully local, no API calls
- **Embeddings** — computed in-process via WASM model
- **Storage** — SQLite at `~/.plumb/memory.db`

You can run Plumb fully offline by using `plumb_remember` for all writes. No LLM API key is needed for the core retrieval and injection pipeline.
